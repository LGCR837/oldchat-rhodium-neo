# OldChat Android 客户端 — 数据模型与存储系统

> 本章分析客户端的数据模型设计、持久化方案和缓存机制。

---

## 一、核心数据模型

### 1.1 Message（消息）

```java
public class Message {
    // 服务端字段
    public String id;                   // 消息唯一 ID
    public String thread_id;           // 会话线程 ID（私聊=双方ID组合，群聊=群ID）
    public String from_uid;            // 发送者 UID
    public String body;                // 消息正文（JSON 字符串，含引用格式）
    public String msg_type;            // 消息类型
    public String media_url;           // 媒体文件 URL
    public String thumb_url;           // 缩略图 URL
    public int duration_ms;            // 音频/视频时长（毫秒）
    public int burn_after_seconds;     // 阅后即焚秒数
    public long burn_start_at;         // 焚毁开始时间戳
    public String burn_preview_text;   // 焚毁预览文字
    public long created_at;            // 创建时间戳
    public int status;                 // 消息状态

    // 本地字段
    public boolean localPending;       // 本地待发送
    public boolean localFailed;        // 本地发送失败
    public String localRequestId;      // 本地请求 ID（用于去重）
    public String localPreviewUri;     // 本地预览 URI
    public int localProgress;          // 上传进度（0-100）
}
```

#### 消息类型常量

| 值 | 说明 |
|----|------|
| `text` | 文本消息（含富文本引用） |
| `image` | 图片消息 |
| `voice` / `audio` | 语音消息 |
| `video` | 视频消息 |
| `emoji` | 表情包消息 |
| `file` | 文件消息 |
| `red_packet` | 红包消息 |

#### 消息状态常量

| 值 | 常量名 | 说明 |
|----|--------|------|
| 0 | `STATUS_NONE` | 无状态/待发送 |
| 1 | `STATUS_SENT` | 已发送 |
| 2 | `STATUS_DELIVERED` | 已送达 |
| 3 | `STATUS_READ` | 已读 |

## 二、持久化存储方案

### 2.1 SharedPreferences

应用重度使用 SharedPreferences 进行数据持久化：

| 文件名 | 存储内容 |
|--------|----------|
| `auth.xml` | 认证信息：access_token, refresh_token, user_id, my_uid, 已记忆的用户名密码 |
| `settings.xml` | 应用设置：主题、通知开关、聊天背景等 |
| `cache.xml` | 缓存元数据 |

使用的操作模式：
```java
getSharedPreferences("auth", Context.MODE_PRIVATE);
```

### 2.2 文件存储

| 路径 | 内容 |
|------|------|
| `getCacheDir()/music/` | 音乐文件缓存（LRU 策略，上限 220MB/120 文件） |
| `getCacheDir()/images/` | 图片缓存 |
| `getCacheDir()/emoji/` | 表情包缓存 |

### 2.3 SQLite

应用可能使用 SQLite 进行消息历史持久化（代码中通过抽象数据访问层访问，具体实现在混淆代码中）。

## 三、数据流分析

### 3.1 消息发送流程

```
用户输入 → 点击发送
    │
    ├─ 1. 构建 Message 对象
    │     • 设置 localRequestId (UUID)
    │     • 设置 localPending = true
    │     • 添加到本地列表（即时显示）
    │
    ├─ 2. POST /direct/send 或 /groups/message/send
    │     • 加密请求体
    │     • 携带 Authorization header
    │
    ├─ 3. 成功 → 更新 localPending = false, status = 1
    │            • 从服务端响应获取 message.id
    │
    └─ 4. 失败 → 标记 localFailed = true
                 • 显示红色感叹号
                 • 点击可重新发送
```

### 3.2 消息接收流程（WebSocket）

```
WebSocket 收到消息
    │
    ├─ 1. WSClient 解析 JSON → WSModels.DirectMessage / GroupMessage
    │
    ├─ 2. Listener.onDirectMessage() / onGroupMessage()
    │
    ├─ 3. 匹配当前 Activity
    │     ├─ 正在当前会话 → 直接追加到 RecyclerView
    │     └─ 不在当前会话 → 增加未读数，更新列表项
    │
    └─ 4. 更新本地存储
```

### 3.3 图片加载流程

```
图片 URL（可能为相对路径 /uploads/xxx.jpg）
    │
    ├─ 1. 拼接 base_url → http://60.205.94.101:8080/uploads/xxx.jpg
    │
    ├─ 2. 检查本地缓存
    │     ├─ 命中 → 直接显示
    │     └─ 未命中 → 异步下载
    │
    └─ 3. 下载 → 缓存到本地 → 显示
```

### 3.4 消息体中的 URL 处理

所有以 `/` 开头的 URL 字段都会自动拼接 `base_url`：

| 字段 | 处理规则 |
|------|----------|
| `avatar_url` | `/avatars/xxx.jpg` → `http://host/avatars/xxx.jpg` |
| `cover_url` | `/covers/xxx.jpg` → `http://host/covers/xxx.jpg` |
| `media_url` | `/uploads/xxx.mp4` → `http://host/uploads/xxx.mp4` |
| `thumb_url` | `/thumbs/xxx.jpg` → `http://host/thumbs/xxx.jpg` |

## 四、消息体格式（v2 引用格式）

消息 body 字段为 JSON 字符串，v2 格式支持引用回复和富文本：

```json
{
    "v": 2,
    "text": "回复的内容",
    "quote": {
        "id": "原消息ID",
        "text": "原消息文本",
        "uid": "原消息发送者UID",
        "name": "原消息发送者显示名"
    }
}
```

纯文本消息的 body 直接为字符串（非 JSON 格式）。
