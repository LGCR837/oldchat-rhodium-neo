## 二、SpongyCastle 加密库

项目使用完整的 `org.spongycastle` 库（Bouncy Castle 的 Android 移植版），包含超过 2000 个类文件。

### 2.2 实际使用的加密操作

```java
// 1. ECDH 密钥对生成
KeyPairGenerator ecGenerator = KeyPairGenerator.getInstance("ECDH", "SC");
ECGenParameterSpec ecSpec = new ECGenParameterSpec("P-256");
ecGenerator.initialize(ecSpec, new SecureRandom());
KeyPair keyPair = ecGenerator.generateKeyPair();

// 2. 共享密钥计算
KeyAgreement ecdhAgreement = KeyAgreement.getInstance("ECDH", "SC");
ecdhAgreement.init(privateKey);
ecdhAgreement.doPhase(serverPublicKey, true);
byte[] sharedSecret = ecdhAgreement.generateSecret();

// 3. AES 密钥派生
MessageDigest sha256 = MessageDigest.getInstance("SHA-256", "SC");
byte[] aesKey = sha256.digest(sharedSecret);

// 4. AES-GCM 加密
Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding", "SC");
byte[] iv = new byte[12];  // 96-bit IV
SecureRandom.getInstance("SHA1PRNG").nextBytes(iv);
GCMParameterSpec gcmSpec = new GCMParameterSpec(128, iv);
cipher.init(Cipher.ENCRYPT_MODE, new SecretKeySpec(aesKey, "AES"), gcmSpec);
byte[] cipherText = cipher.doFinal(plainText.getBytes("UTF-8"));
```

## 三、ECDH 密钥交换协议

### 3.1 握手流程详解

#### 步骤 1：客户端生成密钥对

```
Curve: P-256 (secp256r1)
客户端私钥: sk_client (256-bit)
客户端公钥: pk_client = sk_client * G
```

#### 步骤 2：发送握手请求

```http
POST /auth/handshake
Content-Type: application/json

{
    "client_pub": "BHxh...QYI="     # Base64(客户端公钥)
}
```

#### 步骤 3：服务端响应

```json
{
    "session_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "server_pub": "BFGh...XYZ="     # Base64(服务端公钥)
}
```

#### 步骤 4：派生共享密钥

```
ECDH 共享密钥 = ECDH(sk_client, pk_server)
AES-256 Key  = SHA-256(共享密钥 || session_id)
```

### 3.2 会话密钥存储

```java
public class CryptoUtil {
    private static String sessionId;
    private static SecretKeySpec aesKey;
    private static boolean ecdhSupported;
    private static long encryptDisabledUntil;

    public static boolean shouldEncrypt(String path) {
        if (System.currentTimeMillis() < encryptDisabledUntil) {
            return false;  // 加密失败后的 60 秒降级窗口
        }
        return path != null 
            && !path.startsWith("/auth/") 
            && isEcdhSupported();
    }
}
```

## 四、请求加密/解密流程

### 4.1 加密发送

```java
// 1. 构造 JSON 请求体
JSONObject requestBody = new JSONObject();
requestBody.put("to_uid", uid);
requestBody.put("body", text);

// 2. 序列化为字符串
String plainJson = requestBody.toString();

// 3. AES-GCM 加密
byte[] iv = new byte[12];
secureRandom.nextBytes(iv);
byte[] ciphertext = aesGcmEncrypt(plainJson, aesKey, iv);

// 4. Base64 封装 (IV + 密文)
byte[] ivCiphertext = new byte[12 + ciphertext.length];
System.arraycopy(iv, 0, ivCiphertext, 0, 12);
System.arraycopy(ciphertext, 0, ivCiphertext, 12, ciphertext.length);
String encryptedBody = Base64.encodeToString(ivCiphertext, Base64.NO_WRAP);

// 5. 构建加密请求
JSONObject encryptedRequest = new JSONObject();
encryptedRequest.put("data", encryptedBody);

// 6. 添加安全头
// X-Enc: 1
// X-Session: {sessionId}
```

### 4.2 加密请求的 HTTP 传输

```http
POST /direct/send
Authorization: Bearer {access_token}
Content-Type: application/json
X-Enc: 1
X-Session: a1b2c3d4-e5f6-...

{"data": "base64_encrypted_content"}
```

## 五、安全降级与容错

### 5.1 加密失败处理

```
加密请求发送
    │
    ├─ 成功 → 正常流程
    │
    └─ 失败（HTTP 4xx/5xx 或网络错误）
         │
         └─ 设置 encryptDisabledUntil = now + 60000ms
              │
              ├─ 60 秒内 → 所有请求跳过加密（明文发送）
              │
              └─ 60 秒后 → 重新尝试加密
                            │
                            └─ 成功 → 恢复正常加密
```

### 5.2 Token 刷新安全

```
401 Unauthorized
    │
    ├─ 是 /auth/* 路由？ → 不刷新，直接返回错误
    │
    └─ 否 → POST /auth/refresh { refresh_token }
              │
              ├─ 成功 → 更新 token，重试请求
              └─ 失败 → 计数 +1
                          │
                          ├─ < 5次 → 静默处理
                          └─ ≥ 5次 → 强制重新登录
```

## 六、客户端安全措施

### 6.1 通信安全

| 措施 | 实现 |
|------|------|
| 端到端加密 | ECDH + AES-256-GCM |
| Token 认证 | Bearer token + 静默刷新 |
| 设备绑定 | device_id / IMEI 绑定 |
| 请求防重放 | 每个请求独立 IV (12 bytes random) |
| 阅后即焚 | X-Burn-Secure 头 + 服务端焚毁逻辑 |
| 明文降级限制 | 最多 60 秒明文窗口 |

### 6.2 本地存储安全

| 数据类型 | 存储方式 | 安全性 |
|----------|----------|--------|
| Access Token | SharedPreferences (MODE_PRIVATE) | ⚠️ 明文 |
| Refresh Token | SharedPreferences (MODE_PRIVATE) | ⚠️ 明文 |
| Session Key | 内存（仅运行时） | ✅ 不持久化 |
| 聊天记录 | SQLite（可能为明文） | ⚠️ 未加密 |
| 缓存文件 | 应用私有目录 | ⚠️ 物理访问可读 |

### 6.3 安全管理器

应用包含 `AbstractC0431a` 安全管理器类，当检测到用户切换时自动执行清理：

```java
// 当 SharedPreferences 中的 my_uid 与登录用户不匹配时
if (!savedUid.equals(loginUid)) {
    安全管理器.清理();       // 清除缓存
    AppDataStore.清理();     // 清除本地数据
}
```

## 七、API 加密路由总结

```
所有 API 端点
    │
    ├─ /auth/* 系列 (6 个)
    │    └─ ❌ 不加密（认证路由）
    │

```

## 八、安全局限性分析

| 风险点 | 说明 | 严重程度 |
|--------|------|----------|
| Token 明文存储 | 物理 ROOT 设备可窃取 | ⚠️ 中 |
| 加密降级窗口 | 60 秒明文通信窗口 | ⚠️ 中 |
| 无证书固定 | 未实现 SSL Pinning | ⚠️ 中 |
| 使用 Cleartext | AndroidManifest 允许 HTTP | ⚠️ 高 |
| 本地数据库未加密 | 聊天记录可被直接读取 | ⚠️ 中 |
| SpongyCastle 已过时 | 该库已停止维护 | ⚠️ 低 |
| 混淆过于简单 | ProGuard 重命名可被反推 | ⚠️ 低 |
